# Uplift-моделирование в задаче кредитного риска

Репозиторий магистерской диссертации по применению uplift-моделирования в финтехе. Работа исследует, как выбирать клиентов и канал коммуникации банка (SMS, автоматический звонок, звонок оператора), если цель - снизить вероятность дефолта, а не просто найти самых рискованных заемщиков.

## Идея исследования

Классический кредитный скоринг отвечает на вопрос: **у кого выше риск дефолта?**

Uplift-подход отвечает на другой вопрос: **у кого риск изменится именно из-за коммуникации?**

Эти задачи могут совпадать, если высокий риск связан с высокой полезностью контакта. Но в реальной CRM-задаче это не гарантировано: часть клиентов заплатит без контакта, часть не отреагирует, а для части коммуникация может быть вредной или экономически неоправданной.

В работе сравниваются:

| Подход | Что прогнозирует | Зачем нужен |
|---|---|---|
| Logistic Regression + WoE | риск дефолта | интерпретируемый банковский baseline |
| CatBoost | риск дефолта | сильный нелинейный baseline |
| S-Learner | исход с учетом признака воздействия | простая uplift-постановка на всей выборке |
| T-Learner | отдельные модели исхода для каналов и контроля | гибкая multi-treatment uplift-постановка |
| X-Learner | псевдоэффекты через контрфактические исходы | проверка более сложного meta-learner |
| DR-Learner | doubly robust pseudo-outcome | проверка робастного причинного подхода |
| Oracle strategies | недоступное идеальное знание | верхняя граница качества |

## Данные

Исходная финансовая база - [Home Credit Default Risk](https://www.kaggle.com/competitions/home-credit-default-risk/data). В исходном датасете нет реальной истории коммуникаций, поэтому поверх кредитных признаков построен синтетический причинный слой, имитирующий CRM-политику банка.

Финальная причинная структура:

```text
PD_after = BASE_PD + TRUE_UPLIFT
```

Ключевые переменные синтетического слоя:

| Переменная | Роль |
|---|---|
| `BASE_PD` | базовая вероятность дефолта без коммуникации |
| `CONTACT_PROPENSITY` | вероятность того, что банк инициирует контакт |
| `COMMUNICATION` | фактический канал: `control`, `sms`, `robot_call`, `operator_call` |
| `RISK_SEGMENT` | сегмент риска: `low_risk`, `medium_risk`, `high_risk` |
| `CONTACT_HISTORY` | история предыдущих контактов, задает fatigue effect |
| `PREFERRED_CHANNEL` | предпочтительный канал клиента |
| `INTERACTION_SCORE` | модификатор эффекта по финансовому профилю |
| `DELAY_FLAG` | флаг отложенной реакции |
| `UPLIFT_*`, `PD_*` | потенциальные эффекты и исходы по каналам |
| `TRUE_UPLIFT` | истинный эффект фактически назначенной коммуникации |
| `TARGET_AFTER_CONTACT` | наблюдаемый outcome после коммуникации |
| `ORACLE_*` | oracle-поля для оценки качества, не для обучения |

Ключевые характеристики текущего стенда:

| Показатель | Значение |
|---|---:|
| Наблюдений | 307 511 |
| Столбцов в `uplift-dataset.csv` | 173 |
| Default rate по `TARGET` | 8.07% |
| Default rate по `TARGET_AFTER_CONTACT` | 7.73% |
| Доля contacted / treated | 9.9% |
| Доля control | 90.1% |
| ROC-AUC `BASE_PD -> TARGET` | 0.695 |
| Средний `TRUE_UPLIFT` среди contacted | -0.02522 |
| Leakage-полей, запрещенных для обычного обучения | 22 |

Распределение наблюдаемой политики:

| Канал | Клиентов | mean `BASE_PD` | mean `TRUE_UPLIFT` | target-after rate |
|---|---:|---:|---:|---:|
| `control` | 277 048 | 0.0683 | 0.0000 | 0.0684 |
| `operator_call` | 17 558 | 0.3043 | -0.0347 | 0.2664 |
| `robot_call` | 7 617 | 0.0361 | -0.0177 | 0.0163 |
| `sms` | 5 288 | 0.0098 | -0.0044 | 0.0043 |

## Воспроизводимость

Во всех ноутбуках используется общий seed:

```python
RANDOM_SEED = 91
np.random.seed(RANDOM_SEED)
```

Требования к воспроизводимости:

- запускать ноутбуки сверху вниз из корня репозитория;
- не использовать переменные из предыдущих интерактивных сессий;
- все разбиения и модели со случайностью должны получать `random_state=RANDOM_SEED` или `random_seed=RANDOM_SEED`;
- oracle, treatment, outcome и post-treatment поля не должны попадать в обычные risk-модели.

## Метрики

В проекте используются разные семейства метрик, потому что risk-модели и uplift-модели решают разные задачи.

### Risk-модели

Нормализованный коэффициент Джини:

```text
Gini = 2 * AUC - 1
```

Он показывает качество ранжирования по вероятности дефолта. Для классических risk-baseline моделей сравниваются train, test и OOT.

Результаты на исходной цели `TARGET` в uplift-сравнении:

| Модель | Gini на test |
|---|---:|
| CatBoost, все безопасные признаки | 0.5548 |
| CatBoost, отобранные признаки | 0.4781 |
| LogReg + WoE, все безопасные признаки | 0.5217 |
| LogReg + WoE, 8 отобранных признаков | 0.2930 |

Финальные learning curve по `TARGET_AFTER_CONTACT`:

| Модель | Train Gini | Test Gini | OOT Gini |
|---|---:|---:|---:|
| CatBoost | 49.8 | 42.9 | 42.4 |
| LogReg + WoE | 33.8 | 32.6 | 32.1 |

### Uplift-модели

Для оценки ранжирования по полезности коммуникации используются Qini-кривая и AUUC. В этой постановке полезный эффект имеет отрицательный знак, потому что коммуникация снижает вероятность дефолта, поэтому скор ориентируется на предотвращение дефолта.

Основное сравнение на test:

| Модель / стратегия | AUUC | % от logged-policy oracle |
|---|---:|---:|
| Logged-policy oracle | 413.9207 | 100.0 |
| `BASE_PD` | 365.9255 | 88.4 |
| T-Learner | 359.8117 | 86.9 |
| S-Learner | 335.4985 | 81.1 |
| Counterfactual oracle | 285.2853 | 68.9 |
| CatBoost risk, все признаки | 239.5258 | 57.9 |
| LogReg risk, все признаки | 236.1740 | 57.1 |
| LogReg risk, 8 WoE | 184.4346 | 44.6 |
| CatBoost risk, отобранные признаки | 175.1796 | 42.3 |
| X-Learner | 153.6836 | 37.1 |
| DR-Learner | 141.5144 | 34.2 |
| Случайный выбор | 62.6974 | 15.1 |

Проверка устойчивости:

- 95% bootstrap CI для T-Learner AUUC: `[335.4213; 385.1206]`;
- 95% bootstrap CI для CatBoost risk AUUC: `[220.4326; 260.8380]`;
- paired bootstrap `T-Learner - CatBoost risk`: `+120.3423`, 95% CI `[+103.5537; +137.0560]`, `p ≈ 0.000`.

Проверка по известному истинному эффекту:

| Модель | RMSE к `TRUE_UPLIFT` | Spearman |
|---|---:|---:|
| S-Learner | 0.04985 | 0.2000 |
| T-Learner | 0.13532 | 0.2052 |
| X-Learner | 0.02161 | -0.0023 |
| DR-Learner | 0.02614 | 0.0066 |
| Risk-based `BASE_PD` | 0.15870 | -0.2174 |

### Бизнес-метрики

В `policy_evaluation.ipynb` AUUC переводится в бизнес-язык: предотвращенная вероятность дефолта, экономический эффект, затраты, ROI и доля от oracle.

Сценарные параметры:

| Канал | Стоимость |
|---|---:|
| `sms` | 5 руб. |
| `robot_call` | 30 руб. |
| `operator_call` | 200 руб. |
| Ценность предотвращенного дефолта (LGD-сценарий) | 240 000 руб. |

Сравнение при охвате 20% test-выборки:

| Стратегия | Снижение PD | Экономический эффект, млн руб. | ROI | % от oracle |
|---|---:|---:|---:|---:|
| Случайный выбор | 29.0241 | 6.9658 | 6.0790 | 19.2 |
| Risk-based `BASE_PD` | 106.2274 | 25.4946 | 24.9091 | 70.2 |
| S-Learner | 98.5138 | 23.6433 | 23.0278 | 65.1 |
| T-Learner | 101.6121 | 24.3869 | 23.7834 | 67.2 |
| X-Learner | 42.2132 | 10.1312 | 9.2959 | 27.9 |
| DR-Learner | 45.7842 | 10.9882 | 10.1669 | 30.3 |
| Oracle | 151.2560 | 36.3014 | 35.8917 | 100.0 |

Bootstrap 95% CI снижения PD при охвате 20%:

| Стратегия | Среднее | CI 2.5% | CI 97.5% |
|---|---:|---:|---:|
| Risk-based `BASE_PD` | 106.3294 | 101.6284 | 110.7503 |
| S-Learner | 98.6157 | 94.1737 | 103.0109 |
| T-Learner | 101.7086 | 97.1360 | 106.2540 |
| Oracle | 151.3812 | 146.6740 | 156.5722 |

Multi-treatment policy показала важное ограничение: S-Learner почти всегда выбирает `robot_call` (99.0%), T-Learner почти всегда выбирает `sms` (97.5%), тогда как oracle распределяет каналы разнообразнее: `robot_call` 38.5%, `operator_call` 38.2%, `sms` 21.2%, `control` 2.1%. Это означает, что ранжировать клиентов проще, чем надежно выбирать лучший канал.

## Основные выводы

1. Синтетический стенд содержит нужные причинные свойства: неслучайное назначение контакта, слабый и неоднородный эффект, potential outcomes и oracle-поля.
2. Risk-based стратегии сильны, потому что в текущем стенде риск частично связан с полезностью коммуникации. Поэтому `BASE_PD` может имитировать uplift.
3. T-Learner и S-Learner устойчиво превосходят практические risk-модели по AUUC, но не достигают oracle.
4. X-Learner и DR-Learner не дали преимущества в текущей постановке, что подчеркивает роль overlap, качества propensity-модели и силы сигнала.
5. Бизнес-оценка показывает, что uplift-модель нужно оценивать не только по AUUC, но и по бюджету, ROI, стоимости каналов и устойчивости результата.
6. Multi-treatment выбор канала остается самой сложной частью: для него нужны данные по альтернативным воздействиям и controlled exploration.

## Ограничения

- Коммуникационный слой синтетический, а не наблюдаемый в реальном банке.
- Назначение каналов частично детерминировано риском, поэтому возникает сильный selection bias и проблема overlap.
- `TRUE_UPLIFT` слабый и шумный, что ближе к реальным CRM-эффектам, но усложняет оценку моделей.
- Oracle-поля нужны только для исследования и не доступны в прикладном внедрении.
- Экономические параметры в policy evaluation сценарные и должны заменяться реальными банковскими расчетами.
